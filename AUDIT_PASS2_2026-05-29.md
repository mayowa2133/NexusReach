## üî¥ CRITICAL

### P1. `GET /api/outreach` 500s (`MissingGreenlet`) for normal data ‚Äî Outreach views broken in prod
**Files:** `app/routers/outreach.py:29-51` (`_to_response`), `app/services/outreach_service.py` (`get_outreach_logs` loads only `person`)
`_to_response` reads `log.job`, `log.message`, and `log.person.company` directly off the ORM objects, but `get_outreach_logs`/`get_outreach_log`/`get_outreach_timeline` only `selectinload(OutreachLog.person)`. In async SQLAlchemy, touching an *un-eager-loaded* relationship triggers a lazy load that raises `MissingGreenlet`. `job_id` is set on outreach logs by draft-staging (`_ensure_outreach_log` ‚Üí `_message_job_id`), and `person.company` is set for essentially every discovered contact ‚Äî so the list/detail/timeline endpoints 500 for real users.
**PROOF (live, against Postgres):**
```
listed logs: 1
RESULT: CRASH -> MissingGreenlet | greenlet_spawn has not been called; can't call await_only() here.
```
(created a user+person+job, one outreach log with job_id, called the real `get_outreach_logs` + `_to_response` ‚Üí crash.)
**Why pass 1 missed it:** the backend test harness *mocks the service layer* (per `conftest.py`), so `_to_response` runs against MagicMocks (relationship access returns a mock, never lazy-loads); frontend tests mock the API. The bug only appears against a real DB session.
**Fix direction:** `selectinload(OutreachLog.person).selectinload(Person.company)` + `selectinload(OutreachLog.job)` + `OutreachLog.message` in the three loaders, or stop touching relationships in `_to_response` and pass pre-loaded values.

### P2. Auto-send double-send race ‚Äî recipient can receive the cold email twice
**Files:** `app/tasks/auto_prospect.py:323-365` (`_process_pending_sends`), `app/services/draft_staging_service.py:296-325` (`send_staged_message`), `app/tasks/__init__.py:49` (`task_acks_late=True`)
`_process_pending_sends` `SELECT`s up to 10 `staged` messages with `scheduled_send_at <= now` with **no `FOR UPDATE`/`SKIP LOCKED`/status pre-claim**, then sends each. `send_staged_message` performs the **network send first** and flips `status="sent"` + commits **after**. The only guard is an in-memory `status == "sent"` re-read, which is not durable until after the send. With `task_acks_late=True` + `task_reject_on_worker_lost=True`, a worker restart/redeploy/OOM or the `soft_time_limit=120` firing mid-loop (10 sends √ó 15s timeout = up to 150s) leaves messages `staged` *after they were already sent* ‚Üí the broker redelivers the task ‚Üí those messages are **sent again**.
**PROOF (trace, confirmed by grep):** `task_acks_late=True` present; **zero** occurrences of `with_for_update` / `skip_locked` / `FOR UPDATE` / a transient `sending` status anywhere in `auto_prospect.py` / `draft_staging_service.py`. The send‚Üícommit ordering is in `draft_staging_service.py:296-325`.
**Why it matters:** directly violates the product's safety guarantee; double cold-emails to a recruiter/hiring manager are user-visible spam.
**Fix direction:** atomically claim before sending ‚Äî `UPDATE messages SET status='sending' WHERE id=:id AND status='staged' RETURNING ...` (proceed only if rowcount==1), or `SELECT ... FOR UPDATE SKIP LOCKED` and commit the `sending` transition *before* the provider call.

### P3. ‚ôªÔ∏è Date-sort `::date` cast crashes the whole jobs list on one malformed `posted_at`
**File:** `app/services/job_service.py:1108-1119` ‚Äî the **pass-1 H6 fix** I wrote.
`ORDER BY coalesce(CAST(SUBSTRING(posted_at FROM '^\d{4}-\d{2}-\d{2}') AS DATE), CAST(created_at AS DATE))`. The regex matches date-*shaped* strings that are invalid calendar dates (`2026-13-01`, `2024-02-30`, `0000-00-00`); Postgres raises `date/time field value out of range` on the cast and **aborts the entire query**. `posted_at` is a free-form `String(50)` that is **never validated** (`_coerce_posted_at` only strips whitespace) and many clients pass raw third-party feed/scrape strings (Adzuna `created`, JSearch `job_posted_at_datetime_utc`, Greenhouse `updated_at`, Workable `published`, YC `lastActive`, Wellfound raw HTML `datetime`, Remotive `pubDate`). Hits the **date sort** branch ‚Äî a user-clickable option ‚Äî and **`distance` sort with no resolvable coordinates falls into the same branch**.
**PROOF (live, against Postgres):**
```
SELECT '2026-02-30'::date;                       -> ERROR: date/time field value out of range
-- exact ORDER BY expr over rows incl. posted_at='2026-02-30':
-> ERROR:  date/time field value out of range: "2026-02-30"   (whole query aborts)
```
**Fix direction:** safe parse ‚Äî `to_date(substring(...), 'YYYY-MM-DD')` (lenient, rolls over instead of erroring) wrapped so invalid ‚Üí NULL, or order by `created_at` only, or validate `posted_at` to a real date at ingest.

## üüÝ HIGH

### P4. SSRF ‚Äî exact-job import fetches arbitrary internal/metadata URLs and leaks the content back
**Files:** `app/clients/ats_client.py:1136-1147` (`_parse_generic_exact_url` accepts any http(s) host), `:695-713` (`_fetch_direct_exact_page` ‚Äî `follow_redirects=True`, no host/IP validation), reached from `POST /api/jobs/search/ats` (`job_url`).
The generic adapter is the last in `ATS_ADAPTERS`, so any URL that isn't a known ATS falls through and gets fetched. No allowlist, no private-IP/loopback/link-local block, and `follow_redirects=True` would defeat an allowlist anyway. Crawl4AI/Firecrawl are called with the same URL too.
**PROOF (live):** stood up a localhost-only HTTP service, then drove the real code:
```
parse_ats_job_url accepted internal URL: True | ats_type: generic_exact | host: 127.0.0.1:60942
server FETCHED the internal URL: True
internal content LEAKED back into parsed page title: True
  -> title returned to caller: INTERNAL-SECRET-aws-creds-or-admin-panel
```
**Why it matters:** a user can point `job_url` at `http://169.254.169.254/latest/meta-data/iam/security-credentials/` (cloud metadata / creds), internal admin panels, or `http://localhost:<port>` services and the server fetches them, returning page content. Internal network access + port scan + info disclosure.
**Fix direction:** resolve the host and reject private/loopback/link-local/reserved IPs (incl. on every redirect hop); restrict the import to an allowlist of known job hosts; disable auto-redirects or re-validate each hop.

### P5. Commit-after-send window ‚Üí message re-sent next cycle
**File:** `app/services/draft_staging_service.py:305-325`
Even with a single non-overlapping run: provider send succeeds, then `message.status='sent'` + `await db.commit()`. If the commit throws (pool timeout, DB blip, serialization error) the email was delivered but the row stays `staged` with a past `scheduled_send_at` ‚Üí re-sent on the next `process_pending_sends` cycle.
**Fix direction:** persist an idempotency marker (`provider_message_id` + a durable `sent` claim) in the same transaction as the send-claim, or reconcile against the provider (`check_draft_sent` exists) before re-sending.

## üü° MEDIUM

### P6. Cadence weekly-digest email injects unescaped scraped names (HTML injection)
**File:** `app/services/cadence_digest_service.py:59-66` ‚Äî `f'<td ...>{label}</td>'` where `label` = `person_name`/`company_name`/`job_title` (scraped/user data), **no `html.escape`**. This is the exact bug class fixed for the job-alert digest in pass-1 L11 ‚Äî I missed the sibling `cadence_digest_service._render_html`. `_render_text` is fine.
**Fix direction:** `html.escape()` every interpolated dynamic field (mirror the L11 fix).

### P7. ‚ôªÔ∏è `_refresh_existing_job` wipes `work_mode` to NULL on refresh
**File:** `app/services/job_service.py:302` ‚Äî `job.work_mode = data.get("work_mode")` is **unconditional**, while the pass-1 M1 fix made `location`/geocode/`remote` conditional. `normalize_job_metadata` returns `work_mode=None` for postings with no work-mode signal, so an hourly refresh from a source that yields `None` silently destroys a previously-detected `hybrid`/`remote` label.
**Fix direction:** fold `work_mode` into the same `if value is not None` guard as the other location fields.

### P8. Senior-IC hiring-manager backfill is effectively dead code
**Files:** `app/services/people_service.py:3759-3786` (clone into `hiring_managers`) defeated by `_dedupe_bucket_assignments` (`_match_rank` is sort-key position 0). A verified Staff/Principal peer is `match_quality="direct"`; its hiring-manager clone is `next_best` ‚Üí the peers bucket always wins and the clone is always discarded. The intended "surface a strong IC as HM fallback when the HM bucket is empty" never fires. Pre-existing (not a pass-1 regression).
**Fix direction:** exempt the deliberately-cloned cross-bucket fallback from id-dedup (tag it / give it a synthetic id).

### P9. Outreach create/update store unvalidated `job_id`/`message_id` (missing authorization)
**Files:** `app/services/outreach_service.py` (`create_outreach_log` validates `person` ownership but not `job_id`/`message_id`); `app/routers/outreach.py:150-166` (update passes them through). A user can associate their outreach with another user's `job_id`/`message_id`. (Cross-user *read* via these is currently blocked only by the P1 crash; once P1 is fixed, the unscoped `log.job`/`log.message` relationships in `_to_response` would expose another user's job title.)
**Fix direction:** verify `Job.user_id == user_id` / `Message.user_id == user_id` before persisting (and keep the eager-load scoping when fixing P1).

### P10. `_process_pending_sends` reuses one shared session across all users/messages; error path can poison the loop
**File:** `app/tasks/auto_prospect.py:348-365`. `send_staged_message` commits on the shared `db`; on failure the outer `except` mutates + commits the same possibly-failed-transaction session (`PendingRollbackError` risk), aborting the rest of that user's queue and bleeding into the next user in the fan-out.
**Fix direction:** fresh session per user (or per send); `rollback()` before mutating on the error path.

### P11. Auto-send consent re-check is per-user (batch), not per-message; up-to-5-min disable lag + disconnect-mid-cycle send
**File:** `app/tasks/auto_prospect.py:301-346`. The scheduling gate is correct (`scheduled_send_at` only set under `auto_send_enabled`), and disabling cancels the queue next cycle ‚Äî but a cycle already past the per-user re-check still sends, and `resolve_connected_provider` is resolved once per user so a disconnect after that point still fires the in-flight send (cached token still valid).
**Fix direction:** re-check `auto_send_enabled` + connection immediately before each network send.
